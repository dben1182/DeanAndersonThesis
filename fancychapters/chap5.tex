\section{Safe Flight Corridor Path Following}

Before the final conclusion, it is desirable to combine both of the major contributions of this paper.
These are the trajectory-following controller for a VTOL aircraft and a path-planning algorithm based on RRT to create safe flight corridors. 
A combination of these two contributions is for an eVTOL aircraft to traverse a trajectory generated over an obstacle field by the RRT Safe Flight Corridor algorithm. 
This will test whether the aircraft is capable of following a more complex shape than the simple root trajectories described earlier.

A 2D obstacle field was generated using the RRT algorithm on the North-Down plane. 
The start position was the origin and the end position was $(2000.0, 1000.0)$. 
Nine cubical obstacles of width $100.0$ were placed on a $3 \times 3$ grid. 
The RRT algorithm generated a smoothed SFC path, which is shown in Fig. \ref{fig:SFC_PathFollowing}. 
A video following the aircraft through the trajectory is found at: \href{https://youtu.be/hqRK5J1TDWM}{https://youtu.be/hqRK5J1TDWM}.

\subsection{Controller Failure}

In the previous case, there were no descending sections of the trajectory, and the aircraft was able to follow the trajetory.
Unfortunately, the landing problem, as described in section \ref{sec:landing}, applies to virtually all descending trajectories where a large initial forward velocity is present.
Due to the absence of reverse thrust, the aircraft is unable to stop in its forward velocity.
A map going from an altitude of $1000.0$ meters to $0.0$ meters is shown in Fig. \ref{fig:SFC_PathFollowing_failure}.
A video showing the failed flight may be found at \href{https://youtu.be/eCK1mhtVeUE}{Descent Failure}.
Again, this demonstrates the need for a modification to the control allocation section of the controller to account for this anomaly.

\section{Conclusion}

In this paper, several of the current problems facing robotic development have been addressed, including control, estimation, and planning tools.
Some key points must be reiterated here.

\subsection{Contributions}

This paper has discussed several contributions to general robotics research, with a particular emphasis on VTOL transition control.
The results from this thesis include the following key results:

\begin{itemize}
    \item The transition of a VTOL from hover to forward flight, and vice versa is a very complicated dynamic problem that has been researched extensively over that last several decades.
      Other solutions for VTOL transition usually use some switching criteria based on airspeed to slowly change from hover to fixed-wing flight.
      This paper develops the dynamics and controller for a 2D quadplane VTOL simplification.
      Specifically, it is shown that a 2D quadplane is a differentially flat sytem, so a differentially flat controller is implemented.
      This controller requires an input trajectory of position, velocity, and acceleration for all time samples.
      The pitch-free trajectory tracking controller finds the inertial frame force that will zero the state error.
      The pitch optimizer finds the optimal pitch to generate the most lift possible from the airframe; this causes the switch from hover to fixed-wing.
      The moment controller finds needed moment to control the aircraft to that specified pitch.
      Finally, the elevator and thrust allocators find the control inputs that produced the desired forces and moments.
      This controller is implemented on various reference trajectories, including landing. 
      With the exception of descending, this controller is able to accurately follow a system and transition from hover to fixed-wing flight without the need for gain scheduling.
    \item For a standard fixed-wing aircraft, there are thirty coefficients of aerodynamics, for lift, drag, and east forces, as well as $l$, $m$, and $n$ for roll pitch and yaw moments.
      A recursive LMS filter was developed for each set of coefficients and the corresponding flight data.
      Sensor data is gathered during a flight, including accelerometer data, gyroscope data, airspeed, and angle of attack.
      At each time sample, that data is processed through the each LMS filter and the coefficients are recursively estimated.
      The aircraft is subjected to specific maneuvers that excite phughoid and dutch roll modes, which produce large sensors signals to identify each coefficient.
      These manuevers and LMS filters were implemented in a fixed-wing simulation and proved the potential of aircraft coefficient estimation.
      Future implementations will be used to calibrate VTOL autopilots to have more accurate control.
    \item B-Splines form a basis of robotic trajectory planning. 
      They are extremely useful due to their simplicity, their simple differentiation, and their ability to be parameterized by control points.
      Uniform B-Splines are trajectories created by B-Splines that have no modification to any of the basis functions, including the initial and final points.
      The ability to efficiently generate and update B-Splines trajectories in real time is crucial for many robotic applications.
      B-Splines and their control points can be represented in matrix form, which provides a compact notation that is easily stored and manipulated by a computer.
      This section develops the matrix math necessary to store sampled B-Splines segments in look-up tables for rapid access.
      A library is developed with these look-up tables and minimum-snap trajectories may then be easily generated in real time.
    \item Path planning around obstacles is an important area of research for UAV path planning.
      Rapidly exploring random trees (RRT) is a popular algorithm for path generation through obstacle fields.
      This section develops a variation of the RRT algorithm that generates obstacle-free areas, known as safe flight corridors (SFC), instead of straight lines.
      These rectangular SFCs are placed overlapping in a continuous path from starting to ending points of a trajectory.
      When a complete path SFCs is found, the control points are constrained within an SFC and then optimized to find an optimal path through the SFC path from start to finish.
      This algorithm is demonstrated on a variety of maps in both 2D and 3D.
      Finally, one such 2D path is demonstrated on the 2D VTOL controller described earlier.
\end{itemize}


\section{Future Work}

In the conclusions for each of the preceeding chapters, the future work was already discussed for the respective subject.
Here, we reiterate some of the most important ideas for future work on each subject.

\subsection{Autopilot}

This autopilot framework is still in its infancy.
Much work must be done before this is ever implemented in hardware on a real quadplane.
First, the negative thrust problem must be solved, probably using quadcopter braking methods.
The next step for this algorithm will be to extend from 2D to 3D, which will include proving whether a 3D quadplane is a differentially flat system.
Unfortunately, this autopilot framework is not robust enough for implementation on a quadplane. 
There are too many unknown and nonideal factors that seperate hardware from simulation, especially with the very large angles of attack inherent to transition.
The paper on model predictive control for tiltrotors \cite{mpc_controller} has actually solved many of these problems.
Their implementation is remarkably similar to ours, including optimization for control outputs, model inversion, and roll, pitch, and yaw controllers.
This paper should be studied carefully, compared with this one, and combined to prevent redundant research.
In addition, the uncertainty of the aircraft model during transition requires more robust nonlinear control.
Such nonlinear control for aircraft is discussed in \cite{lavretskyRobustAdaptiveControl2024}; the principles in this book should be studied and implemented for the possibility of future success.
There are also several drawbacks to using the standard quadplane design, including the inefficient use of the lift motors.
It may be wise to begin researching tilt-body design and control \cite{lou_unified_energy_control_tilt_body} due to their efficient use of motors and greater control authority during transition.

\subsection{Aerodynamic Coefficient Estimation}

Aerodynamic coefficient estimation requires many advances and better integration for a complete solution.
Although the LMS filter may be sufficient for simulation, it fails at modelling more advanced linear and nonlinear aerodynamic relationships.
Fortunately, many of these problems have already been resolved in other publications and code repositories. 
For example, \cite{galliker2021datadriven} describes more robust aerodynamic estimation framework, with a working repository \href{https://github.com/ethz-asl/data-driven-dynamics}{data-driven-dynamics}. 
\href{https://github.com/xuhao1/pyAircraftIden}{PyAircraftIden} is another valuable Python library that can perform this estimation much better than our LMS filter does.
Future implementation for estimation should focus on implementing the existing estimation algorithms in real time to assist the VTOL autopilot.

\subsection{B-Spline Libraries}
B-Splines are becoming more important for the UAV trajectory planning.
It has been difficult for the control points for a trajectory to be optimized in real time, especially on an onboard computer during a UAV mission.
With the development of these lookup tables, it is possible for this to change.
However, the current lookup table implementation is only for minimum-snap trajectories.
While these are useful, there are many other parameterizations, such as higher order derivatives, that have not yet been implemented.
Additionally, this work has all been done in Python, which is an interpreted language.
Rewriting these libraries in C++ or Rust may further improve efficiency.

\subsection{SFC RRT}

The safe flight corridor RRT algorithm has been developed for path planning through obstacle fields.
However, it has assumed that such obstacle fields were static, unchanging environments, which is not always the case.
We want to make real-time implementation and updating a possibility for for this algorithm, which is not currently possible because of how slow it currently is.
The algorithm currently runs unconstrained, without any guarantees of convergence or optimal performance.
To overcome this, several existing RRT papers should be reapplied to this application.
For example, \cite{SUN2025103174} develops a more sophisticated method of planning SFCs that morph to encompass most of the available space in the obstacle field.
Their algorithm also incorporates turn constraints directly in.
However, they do not incorporate the same type of B-Spline optimization, which could be an interesting area of future research.
The paper on informed RRT* \cite{Gammell-2014-7921} is currently only applied to straight-line sampling.
Reimplementation of this with SFCs would be a phenominal improvement on the current implementation.
The implementation for this code was all done in Python. It may run significantly faster if it is implemented in C++ or Rust.

\subsection{Software Integration}

When hardware integration for the autopilot is being implemented, the best course of action would be integration with the current software stack.
The BYU Magicc Lab is heavily investing time and resources into the development of open source autopilots developed on the ROS2 framework. 
These autopilot stacks are collectively known as Rosflight \ref{fig:RosflightLogo}. 
Rosflight is developed specifically for developers to easily insert their own code without sifting through massive repositories.
When a better VTOL autopilot is developed, it should be done using Rosflight.

The OpenGL simulation environment we have been using in Python is rather primitive. 
The graphics are rather grainy and unsophisticated and data plots take forever to run.
If this algorithm is integrated into Rosflight, updating these graphics would make a big difference. 
RViz is a visualization software developed specifically for ROS.
Creating the new simulation in RViz would be a great improvement on the current implementation.
